#Personal Agent Level Authorization and Mobile Terminal: Extension + Mobile Dual Terminal and "Thousands of Armies"

Yallet not only has browser plug-ins, but also **Mobile App** (`dev.yallet.mobile-app`, currently mainly RWA). In the future, mobile terminals can pop up notifications and request users to review transactions or any major decisions in real time; at the same time, multiple agents owned by one person can be authorized hierarchically like an organization. In this way, these agents truly become this person's "thousands of troops"**, systematically taking care of all aspects of their life, work, etc.

This article incorporates "dual-end (Extension + Mobile) + push review + personal agent level" into the architecture and vision, and explains how to connect with the existing framework.

---

## 1. Double terminal: Extension + Mobile App

| End | Current Situation | Role in the Agent Economy |
|----|------|------------------------|
| **Chrome Extension** | Complete wallet capabilities (multi-chain, signature, RWA, E2EE) | Policy configuration, automatic execution (free of pop-up windows), interaction with dApp, partial review |
| **Mobile App** | Flutter, currently mainly supports RWA related functions; already has push capabilities (APNS/Firebase, notification_handler) | **Real-time review main battlefield**: push notification → user clicks → approve or reject transactions/major decisions; RWA sending and receiving, Notes, future and extension share the same identity and strategy |

**Design Points**:

- **Same Identity, Same Policy**: Extension and Mobile belong to the same wallet identity (same mnemonic phrase/recovery method or safe synchronization); policies and audits can be synchronized in the cloud, and both ends are consistent.
- **Who does the review**: Extension can pop up the approval page on the desktop; Mobile can reach users in any scenario through **push**, which is more suitable for "real-time, major, and quick response" reviews. Therefore: **regular, small, and within-strategy** can be automatically or pop-up in the extension; **over-limit, first-time, large, cross-border, or upgrade requests from sub-agents** → priority will be pushed to **Mobile** for review by users anytime and anywhere.
- **RWA continuation**: Mobile already supports RWA; Notes, invoices, and commitments generated by Agent can continue to be viewed, confirmed, and signed on Mobile, consistent with the extension experience.

## 2. Push notification: request user Review in real time

**Scenario**: If an agent wants to execute a payment/signing/commitment, but it encounters the conditions "requiring user confirmation" (over budget, new counterparty, single transaction over limit, sub-agent has no permission, etc.), then it **does not execute** directly, but initiates a **review request**, which is sent to the user device via the backend/push service.

- **Mobile**: System push → User click → Open the "Pending Items" page in the App → Display: Who (which agent/scenario), what to do, amount/terms, reason (why you need to confirm) → User chooses **Approve/Reject/Later**.
- **Extension**: If the user is using a browser, they can also pop up the approval page in the extension; if it is not open, they can rely on Mobile push (or email/other channels) to ensure at least one contact.

**Connection with the framework**:

- The existing `requestApproval` is a "synchronous pop-up window"; it can be abstracted into **ReviewRequest**: the same semantics (requires user approval before execution), **transport layer** supports:
- Pop-up window in Extension;
- Or send to "Yallet Push Service" → Push to Mobile (or Extension if online).
- The policy layer can be configured with "**Push to Mobile first** when review is required" or "Extension only" or "Send to both ends" so that you can select the contact method according to the scenario in the future.

## 3. Hierarchical authorization of individual Agents (like organizations)

It is common in organizations today: **Headquarters → Division → Department**, each layer has budget, authority, and approval lines. Isomorphically, a person can also own multiple agents and assign levels and authorizations to them, instead of laying out a bunch of agents that do not belong to each other.

### 3.1 Hierarchical structure (concept)

- **Top layer**: It can be understood as "personal control agent" or "coordination layer" - it does not directly execute specific orders, but does **strategy issuance, budget allocation, and upgrade decisions**. For example: give "life" and "work" a budget cap each month; when the sub-agent exceeds the limit or encounters a "major decision", the request is submitted to the central control or directly to the user (through mobile push).
- **Mid-level**: divided by **field**, for example:
- **Life domain**: supermarket/fresh food agent, subscription renewal agent, housekeeping/payment agent, health-related agent;
- **Work domain**: Invoice/reimbursement agent, contract/commitment agent, freelance payment agent, procurement/price comparison agent.
- **Bottom layer**: An agent that specifically performs certain types of tasks (such as "only responsible for weekly orders for a certain supermarket"), or a special agent provided by the manufacturer (such as a renewal agent for an insurance company).

The **authorization** of each layer to the next layer includes: the available budget (day/week/month), the counterparties that can be dealt with (whitelist), the type of operations that can be performed (payment only/can sign/commit), and the circumstances under which it must be upgraded (over-limit, new counterparty, dispute, or "major decision" are defined by the user).

### 3.2 Upgrade and Review

- **Within policy**: The sub-agent executes automatically within the authorization scope without disturbing the user.
- **Strategy boundary reached**: The sub-agent does not decide on its own, but **initiates an upgrade**:
- Upgrade to **upper-level agent** first (if the upper-level agent has permission and does not exceed the total budget) → the upper-level agent can approve or upgrade again;
- Or directly upgrade to **User**: Generate a **ReviewRequest** and push it to Mobile (or Extension). After the user approves/rejects it, the result is sent back and the sub-agent continues or terminates.
- **"Major decisions"**: can be defined by the user or the master control agent - for example, "a single transaction exceeds

In this way, **level + upgrade** not only ensures that "thousands of troops" can automatically handle a large number of daily tasks, but also ensures that key decision-making rights are always in the hands of people or their designated upper-level agents; mobile push ensures that users can receive and process these key reviews **anytime and anywhere**.

## 4. "Thousands of troops" take care of life and work

Under the above dual-end + push review + hierarchical authorization, it can be embodied as:

- **Life**: Fresh food, subscriptions, payments, health-related payments and commitments, housekeeping appointments and payments, etc., are executed by each agent under the "Life Domain" according to policies; if there are abnormalities or large amounts, users will be asked to review through push notifications.
- **Work**: Invoice classification and payment, reimbursement submission, contract/commitment signing, freelancer milestone payment, procurement price comparison and order placement, etc., are executed by each agent under the "work domain" according to the strategy; similarly, over-limit or major decisions are pushed to Mobile for confirmation by the user.
- **Master Control**: Does not personally execute every order, but sets the budget and rules for each domain, handles cross-domain conflicts, receives "sub-agent undecidable" upgrade requests, and chooses whether to decide by itself (if it is within the master control policy) or to hand it over to the user.

In daily life, users only need to: **Occasionally click on a few pushed reviews** on Mobile, and the rest will be automatically completed by the agent system; if necessary, the level, budget and "what is a major decision" can be adjusted at any time. This is the form in which an agent owned by one person is authorized hierarchically like an organization, collaborates at both ends, and uses Mobile as the real-time review entrance** - the agent truly becomes its "thousands of troops" and systematically takes care of life and work.

## 6. Implementation and product suggestions

1. **Identity and policy synchronization**: Extension and Mobile share the same wallet and policy source; policy editing can be performed on either end, synchronized to the cloud and pushed to the other end, ensuring "consistency on both ends."
2. **ReviewRequest protocol**: Define a unified structure (request id, type, initiating agent, amount/terms, reason, expiration time); the backend/push service is routed to Mobile or Extension according to "user + device"; after the user operation (approval/rejection) is returned, the framework drives execution or cancellation.
3. **Level and Permission Model**: Add "**agent identification + parent agent + budget/whitelist/upgrade rules**" to the policy layer; verify "whether the current operation is within the authorization of the agent" during execution; if upgrade is required, generate a ReviewRequest and optionally go to the "upper agent" first and then the user.
4. **Mobile function expansion**: In addition to the existing RWA and push capabilities, the "pending items" list, ReviewRequest details page and approval/denial operations, optional "policy/level" viewing and simple editing are added, making Mobile a veritable **real-time review and total control portal**.

In this way, **Extension + Mobile dual-end, push-driven real-time review, and personal agent-level authorization** are consistent with the vision of "thousands of troops" and aligned with the existing framework, risk and scenario documents; the specific interfaces (ReviewRequest schema, push payload, hierarchical policy fields) can be further refined in the future.
