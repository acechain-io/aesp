# Yallet Agent framework—vendor access interface definition

Abstract "AI Agentic economy + wallet capabilities" into a pluggable framework: **Yallet provides runtime and capability interfaces, and manufacturers implement policies, message types or external Agent services** to interact with user wallets through this framework.

---

## 1. Framework layering and roles

```
┌─────────────────────────────────────────────────────────────────┐
│Vendor │
│ - Policy Provider │
│ - Message/Protocol Extensions (Message Handlers) │
│ - External Agent service (optional, communicates with wallet through E2EE) │
└─────────────────────────────────────────────────────────────────┘
│ Implement/Call
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│ Yallet Agent Framework (this repository) │
│ - Policy engine: verification, quota, audit │
│ - Execution layer: signature/transfer/commitment (no pop-ups or pop-ups) │
│ - E2EE channel: send and receive encrypted messages, distributed by type │
└─────────────────────────────────────────────────────────────────┘
│ call
                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│ Yallet wallet core (existing) │
│ - Key and unlocking, multi-chain address, balance, Transfer, RWA mint, signature verification │
└─────────────────────────────────────────────────────────────────┘
```

- **Vendor**: Implement "policy source", "processing logic of a certain message type" or run "external Agent service".
- **Framework**: Define interfaces, do policy verification, limits and audits, decide whether to avoid pop-ups or pop-ups, and call the wallet core for execution.
- **Wallet Core**: Not directly open to manufacturers, only used through the capabilities exposed by the framework.

## 2. Core interface definition (TypeScript style, implemented or called by the manufacturer)

### 2.1 Policy Provider — Policy Provider

Vendors can implement this interface and "provide" the user's strategies under the vendor (such as automatic payment rules, entrusted negotiation scope) to the framework.

```ts
// Definition on the framework side, implemented by the manufacturer

/** Strategy type */
type PolicyScope = 'auto_payment' | 'delegated_negotiation' | 'commitment';

/** Single policy (generated by manufacturer or user configuration) */
interface AgentPolicy {
  id: string;
  scope: PolicyScope;
/** Vendor or scene identification to facilitate auditing and permission isolation */
  vendorId: string;
/** Whether to allow pop-up-free execution when conditions are met */
  allowAutoApprove: boolean;
/** Condition: The framework is verified according to this, the specific fields are agreed by scope */
  conditions: PolicyConditions;
/** Optional: validity period, single validity, etc. */
  validUntil?: number;
  createdAt: number;
}

/** Conditional structure by scope (example, extensible) */
interface PolicyConditions {
  // auto_payment
  maxAmountPerTx?: string;
  maxAmountPerDay?: string;
  allowListAddresses?: string[];
  allowListXidentities?: string[];
  token?: string;
  chainId?: string;
  cron?: string;
  // delegated_negotiation
  maxNegotiationAmount?: string;
  allowedCounterparties?: string[];
  // commitment
  maxCommitmentAmount?: string;
  commitmentTypes?: string[];
}

/**
* Policy provider interface.
* The framework will pull the currently applicable policies from the registered Provider and perform merge/priority processing before execution.
 */
interface IAgentPolicyProvider {
  readonly vendorId: string;

  /**
* Get the list of policies applicable to the current wallet and scope under this vendor.
* @param scope - current operation type
* @param context - Optional context (such as counterparty, amount) for manufacturers to perform finer filtering
   */
  getPolicies(scope: PolicyScope, context?: PolicyContext): Promise<AgentPolicy[]>;

  /**
* Notify the framework to refresh when the policy changes (optional, the framework can also poll or pull on demand).
   */
  onPoliciesChanged?(callback: () => void): void;
}

interface PolicyContext {
  origin?: string;
  toAddress?: string;
  recipientXidentity?: string;
  amount?: string;
  token?: string;
  chainId?: string;
  messageType?: string;
}
```

**Framework side responsibilities**:
- Maintain a registry of `IAgentPolicyProvider` (such as `registerPolicyProvider(provider)`).
- Call `getPolicies(scope, context)` before execution, merge according to the framework agreement (such as the union of multiple vendor policies or according to priority), then determine whether `allowAutoApprove` and the conditions pass and are not exceeded, and then decide whether to avoid pop-up windows or pop-up windows.

### 2.2 Execution Request (initiated internally by the manufacturer or framework)

The manufacturer does not directly adjust the wallet core, but initiates an "execution request" to the framework; the framework is responsible for policy verification and routing to/without pop-ups.

```ts
// Definition on the framework side, used internally by the manufacturer or the framework

type ExecutionAction =
  | { type: 'transfer'; payload: TransferPayload }
  | { type: 'sign_personal'; payload: SignPersonalPayload }
  | { type: 'sign_typed_data'; payload: SignTypedDataPayload }
  | { type: 'send_transaction'; payload: SendTransactionPayload };

interface TransferPayload {
  chainId: string;
  token: string; // native | contract address
  toAddress: string;
amount: string; // Minimum unit or human readable, agreed upon by the framework
  memo?: string;
}

interface SignPersonalPayload {
  message: string; // hex or utf8
  chainId?: string;
}

interface SignTypedDataPayload {
  domain: object;
  types: object;
  primaryType: string;
  message: object;
  chainId?: string;
}

interface SendTransactionPayload {
  chainId: string;
  to: string;
  value?: string;
  data?: string;
  gasLimit?: string;
  maxFeePerGas?: string;
  maxPriorityFeePerGas?: string;
}

/** Execution request: with manufacturer and policy information to facilitate auditing and current limiting */
interface AgentExecutionRequest {
  requestId: string;
  vendorId: string;
/** If it is policy-driven automatic execution, fill in the corresponding policyId */
  policyId?: string;
  action: ExecutionAction;
/** Optional: user-visible description, displayed when the window pops up */
  reason?: string;
}

/** Execution results */
interface AgentExecutionResult {
  success: boolean;
  requestId: string;
/** Signature result or tx hash */
  signedPayload?: string;
  txHash?: string;
  error?: string;
  timestamp: number;
}
```

**Framework side responsibilities**:
- Provides `requestExecution(req: AgentExecutionRequest): Promise<AgentExecutionResult>`.
- Internal: Map to policy verification based on `req.action.type` and `scope` → If allowed, avoid pop-up window execution (adjust wallet core), otherwise `requestApproval` pop-up window; the result is returned together with the audit log.

### 2.3 Policy engine - framework side interface (vendor only calls or senses through events)

The policy engine is implemented by the framework, and manufacturers interact with the engine through "registering Policy Provider" and "initiating Execution Request"; the engine is not exposed as a replaceable interface, but the behavior is agreed as follows.

```ts
// Framework implementation and external behavior agreement

interface IAgentPolicyEngine {
  /**
* Register policy provider. The same vendorId can be overridden.
   */
  registerProvider(provider: IAgentPolicyProvider): void;

  /**
* Verify whether the current request is allowed to be executed without pop-up windows by a certain policy.
* Return matching policyId or null (pop-up or rejection required).
   */
  checkAutoApprove(request: AgentExecutionRequest): Promise<string | null>;

  /**
* Record an automatic execution for limit (such as daily accumulation) and auditing.
   */
  recordExecution(requestId: string, policyId: string, result: AgentExecutionResult): Promise<void>;
}
```

Vendors only need to: implement `IAgentPolicyProvider` and register; when initiating `AgentExecutionRequest`, the framework internally calls `checkAutoApprove` and `recordExecution`.

### 2.4 E2EE message channel - receipt distribution and sending (vendor extensible message type)

Delegate negotiation and inter-agent communication rely on E2EE messages. The framework agrees on "message type + payload structure", and manufacturers can register processing logic for a certain type (or only send but not receive).

```ts
// Frame side definition

/** Structured E2EE message (RWA Note or extended protocol body) */
interface AgentMessage {
  id: string;
type: string; // Such as "negotiation.offer", "invoice.request", "commitment.request"
  senderXidentity: string;
  recipientXidentity?: string;
payload: unknown; // Schema agreed by type
  timestamp: number;
threadId?: string; // Session/negotiation thread
}

/**
* Message type handler: The framework calls the corresponding handler when "receiving" a certain type of message.
* Vendors can implement handlers to perform business logic internally (such as updating the UI, calling external Agents, or initiating Execution Requests).
 */
interface IAgentMessageHandler {
readonly messageTypes: string[]; // Such as ["negotiation.offer", "negotiation.accept"]

  /**
* Process received messages. Can return "whether it has been consumed" and optional reply message (sent by the framework).
   */
  handle(
    message: AgentMessage,
    context: MessageContext
  ): Promise<MessageHandleResult>;
}

interface MessageContext {
  walletAddress: string;
  xidentity: string;
  decryptPayload: (encrypted: string) => Promise<unknown>;
}

interface MessageHandleResult {
  consumed: boolean;
reply?: Omit<AgentMessage, 'id' | 'timestamp'>; // The framework will assign id, timestamp and send
}
```

**Framework side responsibilities**:
- In the existing RWA Notes receiving/synchronization path, parse the body as `AgentMessage`; press `type` to find the registered `IAgentMessageHandler` and call it in sequence.
- Provide `sendAgentMessage(message: Omit<AgentMessage, 'id' | 'timestamp'>): Promise<AgentMessage>`, which is sent internally via existing E2EE (such as RWA mint or extended protocol).

**Vendor**: Implement `IAgentMessageHandler` and register it to the framework; optionally define new `type` and payload schema, and declare them in the document to facilitate interoperability with other vendors.

### 2.5 Audit and quota - read-only interface (optional open to manufacturers)

Vendors can only use "execution + policy"; if they need to display or synchronize audit/limit status, they can rely on the read-only interface provided by the framework.

```ts
// Frame side definition

interface IAgentAuditReader {
/** Execution records of a certain strategy within the time range */
  getExecutions(policyId: string, fromTs: number, toTs: number): Promise<AuditEntry[]>;

/** The used limit of a certain strategy on the day (such as amount, number of times) */
  getUsageToday(policyId: string): Promise<UsageSnapshot>;
}

interface AuditEntry {
  requestId: string;
  policyId: string;
  vendorId: string;
  action: ExecutionAction['type'];
  result: AgentExecutionResult;
  timestamp: number;
}

interface UsageSnapshot {
  policyId: string;
  amountByToken: Record<string, string>;
  count: number;
}
```

## 3. How the framework is exposed to manufacturers (SDK/runtime)

### 3.1 "Framework runtime" within extensions

- The framework code and the wallet core are in the same warehouse and built into the same extension.
- The extension is initialized in background or offscreen:
- `IAgentPolicyEngine` (including registry);
- The policy engine pulls policies from all registered `IAgentPolicyProvider` and verifies them every time `requestExecution` is executed;
- The execution layer removes pop-ups or pop-ups based on the verification results and calls the wallet core (Transfer, Sign, SendTx).
- In the E2EE receiving path, after parsing out `AgentMessage`, distribute it to the registered `IAgentMessageHandler` according to `type`.

### 3.2 How to connect manufacturers

- **Method A: Vendor’s own extension/application**
- Vendors implement `IAgentPolicyProvider` and/or `IAgentMessageHandler` in their own extensions or web pages.
- Communicate with Yallet extension through "Yallet Agent SDK": SDK internally uses `chrome.runtime.sendMessage` or Native Messaging to communicate with Yallet, calling the interfaces exposed by the framework (such as `registerProvider`, `requestExecution`, `sendAgentMessage`).
- Prerequisite: The user has installed the Yallet extension; the message protocol needs to be agreed between the manufacturer's extension and Yallet (see below).

- **Method B: "Vendor Module" built into Yallet**
- Vendors provide Provider / Handler implementations in the form of npm packages or configurations; Yallet loads and registers them when building or running.
- Suitable for official cooperative manufacturers, with unified experience; updates need to be released with Yallet or use a hot update mechanism.

### 3.3 Yallet ↔ Manufacturer’s Message protocol (mode A)

If the manufacturer is in an independent extension/process, the following message can be agreed with Yallet (example):

```ts
// From manufacturer to Yallet
type ToYalletMessage =
| { method: 'AgentFramework.registerProvider'; params: [/* Serialized provider description or port */] }
  | { method: 'AgentFramework.requestExecution'; params: [AgentExecutionRequest] }
  | { method: 'AgentFramework.sendMessage'; params: [Omit<AgentMessage, 'id'|'timestamp'>] };

//Send from Yallet to manufacturers (such as policy pull, message push)
type FromYalletMessage =
  | { method: 'AgentFramework.getPolicies'; params: [PolicyScope, PolicyContext?]; id: string }
  | { method: 'AgentFramework.onMessage'; params: [AgentMessage] };
```

Provider is implemented on the manufacturer side. When Yallet pulls policies through `getPolicies`, the manufacturer side responds; or the manufacturer establishes a long-lived port through `chrome.runtime.connect`, and Yallet sends `getPolicies` and receives policies through the port. For details, you can customize another version of the "Yallet Agent SDK Agreement" document.

## 4. Interface summary table (manufacturer’s perspective)

| Interface | Vendor implementation/call | Description |
|------|------------------|------|
| `IAgentPolicyProvider` | Vendor **Implementation** | Provides a policy list for the vendor; the framework pulls and verifies it. |
| `AgentExecutionRequest` + `requestExecution` | Manufacturer **call** | Initiate transfer/signature/transaction request; the framework decides whether to pop-up or avoid pop-up and execute. |
| `IAgentMessageHandler` | Vendor **Implementation** | Process certain types of E2EE messages received, optionally replying. |
| `sendAgentMessage` | Vendor **Call** | Send E2EE messages as the current wallet identity. |
| `IAgentAuditReader` | Vendor **Call** (optional) | Check audits and limits, for UI or compliance. |
| `IAgentPolicyEngine.registerProvider` | Vendor **call** | Register your own Policy Provider. |

## 5. Recommended directory and package structure (framework code)

```
extension/
  public/
    agent-framework/
types.ts # Centralized definition of the above interfaces and types
policy-engine.ts # IAgentPolicyEngine implementation
execution.ts # requestExecution, pop-up-free/pop-up routing, auditing
messaging.ts #AgentMessage parsing, sendAgentMessage, Handler registration and distribution
audit.ts # IAgentAuditReader implementation
index.ts # External export and registerProvider/registerMessageHandler
```

If manufacturers access it in the form of npm packages, they can use a separate package such as `@yallet/agent-sdk`, which relies on the above types and the message protocol. It does not rely on the wallet core implementation and only communicates with the extension through messages.

---

## 6. Version and compatibility

- The interface uses **Semantic Version**; new optional fields or new `ExecutionAction` types are considered minor; deletions or required changes are considered major.
- The manufacturer declares the supported framework API version (such as `agentFrameworkVersion: '1.0'`) when registering, and the framework can make it compatible or reject incompatible versions.
